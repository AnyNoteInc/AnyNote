# Спецификация: публичная страница «История изменений» (`/changelog`)

**Дата:** 2026-05-26
**Статус:** на ревью
**Автор:** brainstorming-сессия

## Цель

Публичная (без авторизации) страница `/changelog`, на которой пользователи видят
понятную историю развития продукта: что нового появилось и что заметно
улучшилось, по релизам, сверху — самое свежее. Источник правды — markdown-файл
`docs/changelog.md`, который рендерится на страницу.

## Решения (зафиксированы в брейнсторминге)

1. **Группировка — по минорным версиям** (~19–21 блок). Патч-релизы (`1.0.x`,
   `1.15.x` …) сворачиваются в свою минорную версию. Один блок на `1.1`, `1.2`,
   … `1.21`, плюс крупный блок `1.0` (запуск).
2. **Охват — только пользовательское.** Показываем фичи и заметные улучшения,
   которые пользователь видит/ощущает. Чисто технические коммиты (CI, деплой,
   тесты, lint, SonarCloud, рефакторинг сборки) не показываем.
3. **Блок «Готовится» сверху** — функции, уже влитые в `main`, но ещё без тега
   релиза: комментарии к страницам и публичные ссылки с правами доступа.

## Архитектура

Точная калька с существующих публичных страниц `/pricing` и `/terms`. Никакого
нового механизма рендеринга — используем уже настроенный MDX-конвейер
(`@next/mdx` + `remark-gfm`, алиас `@docs`, стили в
[`apps/web/src/mdx-components.tsx`](../../../apps/web/src/mdx-components.tsx)).

## Формат `docs/changelog.md`

- **Без верхнего `# H1`.** В отличие от файлов `docs/terms/*.md` (там H1 есть),
  здесь заголовок страницы даёт `PublicPageShell` (`title="История изменений"`),
  а лид-абзац — его `description`. Дублировать заголовок в файле не нужно —
  файл начинается сразу с блока «Готовится».
- Один блок на версию. Заголовок версии — `##` (рендерится как заметный
  подзаголовок). Формат: `## v1.18 — Канбан-доски · 16 мая 2026`.
- Внутри блока — подзаголовки `**Новое**` и `**Исправлено**` (жирный текст, не
  отдельный markdown-heading, чтобы не плодить уровни) и маркированные списки.
- Между блоками — горизонтальный разделитель `---` (рендерится как `<Divider>`).
- Порядок — новейшее сверху. Блок «Готовится» — самый первый.

Пример:

```markdown
## Готовится

Функции уже доступны в продукте и стабилизируются перед официальным релизом.

**Комментарии к страницам**
- Обсуждения прямо в тексте: выделите фрагмент и оставьте комментарий…

---

## v1.21 — Избранные чаты и обновлённые пространства · 19 мая 2026

**Новое**
- Избранные чаты — быстрый доступ к важным диалогам с ИИ.
- Переработанный экран рабочих пространств.
```

## Контент: маппинг версия → пользовательское описание

Канонический черновик. При реализации формулировки могут чуть шлифоваться, но
состав блоков и решения «что показать / что опустить» — отсюда.

### Готовится (в `main`, без тега)

**Комментарии к страницам**
- Обсуждения прямо в тексте: выделяете фрагмент — оставляете комментарий, ведёте
  тред, помечаете «решено» и открываете заново.
- Упоминания участников через `@` с уведомлением упомянутому.
- Всплывающее окно треда у выделенного фрагмента и прямые ссылки на конкретный
  комментарий (`#comment-…`).
- Комментарии работают и на публичной странице по ссылке.

**Публичные ссылки с правами доступа**
- Делитесь страницей по ссылке; роль «Комментатор» даёт чтение и комментарии без
  права на правку.
- Гостевой (анонимный) доступ для тех, у кого нет аккаунта.

## Тестирование

- **E2E (Playwright, `apps/e2e/`):** новый спек или дополнение существующего:
  1. `/changelog` открывается анонимно (без логина), отдаёт 200.
  2. На странице виден заголовок «История изменений» и хотя бы один блок версии
     (например, текст «Канбан-доски» / «v1.0»).
  3. В шапке главной (`/`) есть пункт «Обновления», ведущий на `/changelog`.
- **Гейты:** `pnpm --filter web check-types`, `pnpm lint`, `pnpm --filter web build`
  (на статической сборке поймает RSC↔Client-ошибки). Затем `pnpm gates`.
- **Ручная проверка:** `pnpm --filter web dev`, открыть `/` → клик «Обновления»,
  проверить рендер, светлую/тёмную тему, мобильную ширину.

  собирается и в dev (Turbopack), и в prod (`next build`, webpack) — как уже
  работает для `docs/terms/*.md`.

## Вне области

- Автогенерация `docs/changelog.md` из git-тегов/коммитов — здесь файл пишется
  вручную (курируемый, человеко-читаемый текст). Авто-пайплайн — отдельная задача.
- RSS/Atom-лента изменений — не в этой итерации.
- Подписка на обновления по email — не в этой итерации.
