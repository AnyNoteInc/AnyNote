# anynote — Deep Audit: Research & Prioritized Improvement Backlog

> Synthesis of an adversarially-verified multi-agent audit (42 kept findings; rejected/already-mitigated items removed). This is a **research + backlog** document, not an execution plan — pick items from §3 as you see fit. Every backlog row preserves its `file:line` anchor; those anchors are the point.
>
> **Method.** A recon scout mapped the monorepo, then six auditors (Database/Prisma, Web/RSC, Realtime & AI/RAG, Algorithms & Patterns, Storage Systems, Architecture & Jobs) read the real code in parallel. Every finding was then handed to an adversarial verifier prompted to *refute* it — checking whether the problem is real, already mitigated, or over-rated. 45 raw findings → 42 kept after verification.
>
> **Badge note.** Where the verifier corrected an auditor's original rating, the corrected value is used throughout (e.g. several "high" impacts were honestly downgraded to medium/low after on-disk checks). Original ratings are noted only where the gap is instructive.

## 1. Executive summary — the biggest opportunities

**1. The AI chat request is the single most expensive path, and it does unbounded work twice over.** `buildChatHistoryMessages` (`apps/web/src/lib/chat/chat-history.ts:75-106`) walks the chat-ancestor chain with one sequential `chat.findFirst` per node (up to 50), then runs one `chatMessage.findMany` **per ancestor chat with no `take`** — loading every DONE message (including heavy `parts` JSON) only to slice to the last 4–10 in JS. Separately, the LangGraph executor (`apps/agents/.../nodes/executor.py:55-86`) re-serializes the full chat history + all attachments (up to ~512 KB) into the system prompt and re-sends the entire growing transcript on **every** tool-loop step, with no prompt-caching and no explicit recursion/tool-call budget. The win is to bound the message fetch with an indexed keyset window, and to mark the static prefix with provider prompt-caching + truncate model-visible tool results. These are the highest-leverage DB-load and token-cost reductions in the codebase.

**2. There is no cache or KV tier anywhere, and the few places that need one are starting to hurt.** The chat SSE stream registry is a process-local `Map` singleton (`apps/web/src/lib/chat/active-stream-registry.ts`), which is silently incorrect the moment `apps/web` runs more than one replica — a reconnect landing on a different process returns a premature `DONE`. `getWorkspaceFeatures` recomputes 2 queries on every protected render and every search. There is no rate limiter on the one genuinely unthrottled abuse-surface route (`/api/agents/generate`), and better-auth does a Postgres session lookup on every uncached request (no `cookieCache`, no `secondaryStorage`). A single Redis/Valkey instance is the one storage-technology addition worth green-lighting — but only if multi-replica `apps/web` is actually a goal; today's deploy is single-replica, so these are latent, not live, defects.

**3. Page ordering picked the worse of two designs that already coexist in this repo.** Pages use a `prevPageId @unique` singly-linked list (`schema.prisma:516`), forcing a 3–5 row shuffle per drag under the unique constraint and a documented cross-collection "drag-to-sort-invisible" bug (the chain is scoped by `parentId` but not `collectionId`). The Kanban subsystem in the same repo already solved this correctly with a `position Float` column and `positionBetween` — a reorder is one single-row UPDATE, and the pattern is reused across 7 models. Migrating pages to fractional indexing collapses the multi-row shuffle, kills the client-side O(n²) `flattenTree` rebuild, and eliminates the cross-collection bug by construction. High blast radius (~30 referencing files), so it's a deliberate, larger investment.

**4. Qdrant is run without payload indexes, and every page save re-embeds the whole document.** Collections are created with only `VectorParams` — no `create_payload_index` on `workspaceId`/`pageId` anywhere (`vector_store_repository.py`), so the per-tenant filter on every RAG query and every per-page delete is an unindexed payload scan over a collection that holds **all** workspaces sharing an embedding model. Compounding this, the indexer re-embeds every chunk on every save with no content-hash dirty check (`vectorization-cron.service.ts` + `vectorize_page.py`), so editing one paragraph re-runs paid embedding calls over the entire document. Two cheap, mostly-independent fixes — add keyword payload indexes; gate vectorization on a `(content, provider, model)` hash.

**5. The Page-listing visibility predicate scans the tenant's pages with no matching index, and search pays a second query to re-apply it.** `listByWorkspace` (runs on every protected render) filters `workspaceId + archivedAt:null + deletedAt:null + isTemplate:null` plus a 4-arm visibility OR and a database-row exclusion, but no composite/partial index matches it (`schema.prisma:565-572`). Full-text and RAG search each then issue a **second** `page.findMany` purely to re-apply the same visibility OR (`page-search.ts:91-116, 238-249`) — and the FTS `LIMIT` is applied *before* visibility filtering, a latent under-return correctness gap. A partial index plus inlining the (already partly-inlined) visibility predicate into the FTS SQL addresses both, but the index column choice needs an `EXPLAIN`-driven decision and the inlined predicate forks a single-source-of-truth that needs a guard test.

**6. The RSC → client data path re-fetches over HTTP what the server already computed.** The `(active)` layout resolves role, page tree, and features server-side, then the client re-queries them: `getMyRole` runs server-side *and* as a client `useQuery`; `listByWorkspace` is fetched server-side then force-refetched client-side via `staleTime:0`. There is **zero** React Query hydration (`HydrationBoundary`/`dehydrate`/`prefetch`) anywhere in `apps/web`. None of this blocks first paint (props already render), but it is redundant post-hydration round-trips on every navigation. The cheap, correct fix is prop-drilling already-resolved values + seeding `initialData`; the full dehydrate pattern needs plumbing this architecture (`createTRPCReact` + `createCaller`, separate QueryClients) doesn't yet have.

**7. The cron/background tier is where the architecture is weakest — duplicated, unisolated, and one silent money-path gap.** The billing renewal cron is the **only** `@Cron` with no try/catch + `Sentry.captureException`, and `SentryGlobalFilter` doesn't catch crons — so a DB blip in the renewal batch-claim vanishes from ops on the money path. Outbox claim+backoff logic is copy-pasted across indexer/webhooks/telegram/notifications (telegram is a byte-for-byte copy of webhooks). The MCP request-server shares one process and one replica with 7 crons, 3 of which poll Postgres every 5 s, with no module toggle — so MCP can't scale horizontally without multiplying the poll floor. And the indexer resolves embedding-provider credentials via a path that **ignores `connectionEnc`**, so workspaces with encrypted credentials silently never index.

**8. A latent silent-failure on encrypted credentials is the one finding that behaves like an outright bug.** `indexer-connectionenc-divergence`: the indexer cron selects only plaintext `provider.connection` and passes it to `parseAiProviderConnection`, which **throws** on the empty object that every workspace-configured provider now stores (real creds live in `connectionEnc`). The page FAILs after 5 outbox attempts and never indexes; RAG just returns fewer results, so it's invisible. Web's two other resolvers decrypt `connectionEnc` first — the indexer is the lone divergent path. Small, copyable fix; the only finding rated genuinely **high** impact post-verification.

## 5. Cross-cutting themes

- **No cache/KV layer anywhere.** The same absence produces: a process-local SSE registry (S1), recomputed plan features (S2), per-request session DB lookups + no rate-limit substrate (S3), per-message MCP discovery (R7), and the indexer's per-save re-embed (R1, a content-cache miss). One Redis tier addresses 4 of these; the others want in-process or content-addressed caches.
- **Outbox/queue infra is duplicated 3–4×.** Indexer, webhooks, telegram (byte-for-byte), and notifications each re-implement claim + `LEAST(300,POWER(2,...))` backoff (X4), and the 5 s poll loop is itself triplicated (X5). The shared boundary was drawn one layer too shallow (telegram imports webhook's *visibility* helpers but copies its *claim* machinery). None have lock-staleness reclaim — a latent stuck-row class.
- **Provider-credential resolution forks repeatedly, and it has already shipped bugs.** The `connectionEnc`-vs-plaintext precedence is the recurring foot-gun: it caused the page-search `workspaceId`-gate bug (in memory), the generate-path bug, and now the silent indexer failure (X1). Single-sourcing it (A6 + X1) is both a cleanup and a correctness fix.
- **The DB does unbounded work on the two hottest paths.** Chat generation (D1/A2) and the page sidebar listing (D2) both scan/transfer in proportion to conversation/workspace size with no `take` and no matching index — the dominant scaling risk. Same shape recurs in BOARD grouping (D4) and getById (D5).
- **Sequential ancestor/tree walks where a CTE or walk-up wins.** The same anti-pattern appears in chat history (D1), guest grant resolution (D6), and the reorder ancestry guard (A5) — each loads more than it needs by descending or stepping one row at a time.
- **RSC computes data the client then re-fetches.** No React Query hydration anywhere (W1/W2/W3/W4); `getMyRole` and `listByWorkspace` are resolved server-side and re-queried client-side. Low-risk to dedup via props/`initialData`; the full dehydrate pattern needs plumbing this stack lacks.
- **Crons are the architecturally weakest tier.** Observability gap (X2), duplication (X4/X5), and no isolation/scaling boundary (X3) all cluster in the background workers, not the (well-enforced) package graph.
- **The good news, recurring.** Read-stack batching (`augmentRows`, no N+1), keyset pagination for table views, `FOR UPDATE SKIP LOCKED` claims, idempotent Qdrant point IDs, GIN FTS, the dependency-cruiser layering, and `@repo/domain` as a real shared write-authority are all genuinely well-built and were left alone.

## 6. What we deliberately did NOT recommend

Rejected and already-mitigated findings were filtered out by the adversarial verification pass before this document. A few tempting-but-wrong ideas are called out explicitly so future readers don't re-propose them:

- **A dedicated search engine, OLAP/columnar store, time-series DB, graph DB, or edge KV** — all evaluated and rejected (§4). Postgres + Qdrant + S3 is the right split here; the search gaps are indexing/predicate issues, not a missing engine.
- **A keyframe+delta revision encoding for PageRevision (R8)** — tempting for storage, but Postgres TOAST already compresses the blobs and the delta scheme risks orphaning deltas when the prune cron deletes interior revisions. Deferred, not adopted.
- **A blanket Redis read-through cache for `getWorkspaceFeatures` (S2)** — the right-sized fix is folding two queries into one or parallelizing the existing layout batch; standing up Redis purely for this would be the wrong-sized solution given the 4-place env pipeline cost.
- **Adding `@mui/material`/`@mui/icons-material`/`@mui/x-charts` to `optimizePackageImports` (W8)** — a no-op: those barrels are already deep-imported everywhere. Only the first-party `@repo/ui` barrels are worth adding.
- **"Skip the redirect" in the page/chat route active-workspace check (W6)** and **the in-process throttle/last-write cache in yjs (R9)** — both look like easy wins but break shell consistency / multi-replica correctness respectively; only the `Promise.all` parallelization parts are safe.
- **Treating `apps/yjs` as a `@repo/domain` layering violation (X6)** — it isn't; the architecture doc permits Tier-5 presentation to write against pure adapters, the outbox contract is already single-sourced in `@repo/db`, and the throttle divergence is intentional.
