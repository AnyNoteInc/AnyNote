# Визуальное выделение родительских задач и блок «Подзадачи»

**Дата:** 2026-06-06
**Статус:** Дизайн утверждён, готов к написанию плана

## Контекст и проблема

В текущей реализации Kanban визуальное отличие родительских задач от подзадач
недостаточно очевидно — особенно при большом объёме работ на доске, в таблице и
на диаграмме Ганта. Пользователю приходится «вчитываться» в иерархию, чтобы
понять структуру. Кроме того, находясь внутри родительской задачи, нет быстрого
доступа к списку её дочерних элементов и их статусов.

## Цель

Сделать иерархию задач очевидной с первого взгляда во всех трёх представлениях
(доска, таблица, Гантт) и обеспечить быстрый доступ к прогрессу подзадач из
детального просмотра.

## Определения

- **Родительская задача** — задача, у которой есть хотя бы одна дочерняя
  (`children.length > 0`, т.е. существует другая задача с `parentId === task.id`).
  Одиночные top-level задачи без детей **не** считаются родительскими и не
  выделяются.
- **Подзадача (дочерняя задача)** — задача с непустым `parentId`.
- **Статус задачи** — определяется видом колонки (`KanbanColumn.kind`:
  `ACTIVE` / `DONE` / `CANCELLED`), к которой принадлежит задача. Цвет статуса —
  `column.color` при наличии, иначе дефолт по `kind`.

## Принципиальное решение по данным

`kanban.board.getBoard` уже возвращает **плоский** список всех задач страницы
со всеми нужными полями (`parentId`, `columnId`, `startDate`, `dueDate`,
`assignees`, `labels`). Все производные данные иерархии вычисляются **на
клиенте**, без изменений в бэкенде, Prisma-схеме или tRPC.

Все три вью и детальный просмотр уже потребляют один и тот же объект `board`
(результат `getBoard`), поэтому карта детей строится один раз на уровне каждого
вью (мемоизированно) и пробрасывается вниз.

**Изменений в `packages/db`, `packages/domain`, `packages/trpc` — нет.**

## Архитектура

### Новый чистый хелпер: `kanban/lib/hierarchy.ts`

Рядом с существующим `kanban/lib/assignees.ts`. Framework-agnostic, покрыт
unit-тестами.

```ts
// Группировка задач по parentId. Один проход.
buildChildrenMap(tasks: BoardTaskData[]): Map<string, BoardTaskData[]>

// Дети конкретной задачи (пустой массив, если детей нет).
getChildren(map: Map<string, BoardTaskData[]>, taskId: string): BoardTaskData[]

// Прогресс: доля детей со статусом DONE.
// Возвращает { total, done, ratio } — ratio в [0, 1].
subtaskProgress(
  children: BoardTaskData[],
  columns: BoardColumnRow[],
): { total: number; done: number; ratio: number }
```

`subtaskProgress` определяет «выполнено» по `column.kind === 'DONE'`
ребёнка (через lookup `columnId → column`). Отменённые (`CANCELLED`) в числитель
не входят, но входят в знаменатель `total` (общее число подзадач).

### Новый общий компонент: `kanban/components/parent-badge.tsx`

Компактный бейдж родительской задачи, используется и на доске, и в таблице, для
визуальной консистентности.

```tsx
<ParentBadge count={n} />
```

- `AccountTreeIcon` (MUI) + число детей.
- Нейтральная палитра (`action.hover` фон, `text.secondary` текст), высота ~20px,
  скруглённый, чтобы не конфликтовать с priority-чипом.
- `Tooltip`: «Родительская задача · N подзадач» (с корректной формой
  множественного числа на русском: подзадача / подзадачи / подзадач).

## Реализация по представлениям

### 1. Доска — `board-card.tsx` + `board-card-model.ts`

- В `board-card-model.ts`: добавить в модель поле `childCount: number`,
  вычисляемое из переданной карты детей (карта строится один раз в
  `board-view.tsx` и пробрасывается, чтобы не пересобирать на каждой карточке).
- В `board-card.tsx`, если `childCount > 0`:
  - Показать `<ParentBadge count={childCount} />` в ряду type/priority-чипов.
    Условие отрисовки ряда расширяется на
    `model.type || model.priority || childCount > 0`, чтобы бейдж появлялся даже
    без type/priority.
  - Заголовок задачи: `fontWeight={700}` вместо `600`. Подзадачи и одиночные
    задачи остаются `600`.

### 2. Таблица — `table-view.tsx` + `sprint-section.tsx`

- В `table-view.tsx`: построить карту детей один раз (мемоизированно, рядом с
  `grouped`) и пробросить в `SprintSection` → `TaskRow`, как уже пробрасывается
  `allTasks`/`columns`.
- В `TaskRow` (внутри `sprint-section.tsx`), если у задачи есть дети:
  - Слева от заголовка — `<ParentBadge count={childCount} />` (тот же компонент,
    что на доске).
  - Заголовок родителя — `fontWeight={600}` (обычные строки `body2` без
    жирности остаются normal).

### 3. Гантт — `gantt-view.tsx`

- Сохранить `type: 'task'` для всех задач (не менять drag-поведение, не требовать
  чтобы родитель «накрывал» даты детей).
- Сохранить существующие зависимости `dependencies: t.parentId ? [t.parentId] :
  undefined` — связь родитель→ребёнок уже рисуется стрелкой.
- Для задач с детьми использовать **насыщенный вариант** палитры той же статусной
  семьи (вместо `COLUMN_COLORS[kind]`). Ввести `PARENT_COLUMN_COLORS` — более
  тёмный/насыщенный `bg` и `selected` для каждого `kind` (пример:
  ACTIVE `#1d4ed8` вместо `#3b82f6`; DONE `#15803d` вместо `#22c55e`;
  CANCELLED `#6b7280` вместо `#9ca3af`). Выбор палитры — по наличию детей у задачи.
- **Штриховка/паттерн** полосы — вне рамок штатного API библиотеки
  (`gantt-task-react` красит только цветом). Помечено как опциональная доработка
  на будущее (потребовал бы хрупкого CSS-оверлея по внутренней SVG-вёрстке).

### 4. Детальный просмотр — новый `kanban/task/subtasks-section.tsx` + `task-detail-modal.tsx`

Новый компонент `SubtasksSection`, рендерится в `TaskForm` **сразу под блоком
описания, перед блоком вложений**, и **только если у задачи есть дети**.

Состав блока:

- **Заголовок** «Подзадачи» + справа счётчик «выполнено N из M».
- **Прогресс-бар**: `LinearProgress` (determinate), процент =
  `subtaskProgress(...).ratio * 100`. Стандартный success/primary цвет.
- **Список подзадач** — каждая строка:
  - **Цветной индикатор статуса** — маленький кружок, цвет из колонки ребёнка
    (`column.color` или дефолт по `kind`).
  - **Название** подзадачи.
  - **Текст статуса** — название колонки (напр. «В работе» / «Готово»),
    приглушённым `text.secondary`.
  - **Кликабельна**: открывает детальный просмотр подзадачи через смену
    `?taskId=` тем же `router.replace`, что используют карточки/строки — модалка
    перерисовывается на новый `taskId`. Никаких новых запросов.

Данные: `getChildren(map, task.id)` либо `board.tasks.filter(t => t.parentId ===
task.id)`. `board` уже доступен в `TaskForm`.

## Изоляция и тестируемость

- `hierarchy.ts` — чистые функции, тестируются изолированно (unit-тесты на
  `buildChildrenMap`, `getChildren`, `subtaskProgress`, включая краевые случаи:
  нет детей, все DONE, смешанные статусы, наличие CANCELLED в знаменателе).
- `ParentBadge` и `SubtasksSection` — презентационные компоненты с явными
  пропсами; внутреннюю логику можно менять, не затрагивая потребителей.
- Все остальные изменения — точечные правки рендера в существующих компонентах.

## Что НЕ входит в объём (YAGNI)

- Изменения бэкенда / Prisma / tRPC.
- Штриховка/паттерн полос на Гантте (опционально, на будущее).
- Многоуровневая (вложенная глубже одного уровня) визуализация иерархии.
- Сворачивание/разворачивание поддерева на доске или в таблице.
- Drag-and-drop назначения родителя из списка подзадач.
